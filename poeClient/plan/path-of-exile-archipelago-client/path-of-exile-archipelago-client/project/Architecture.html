Apworld v — see /Version.py

Stack: Electron 30 + TypeScript (strict) + React 18 + Vite. UI state in Zustand. Archipelago protocol via archipelago.js. Global input via uiohook-napi. Foreground-window check via active-win. File watching via chokidar. OAuth PKCE in a sandboxed BrowserWindow. Packaged with electron-builder (NSIS + AppImage).

Why this matters for design: the Python client is a headless background service talking to the user through in-game whispers because that was the only surface it could reach. A real UI means !gear, !gems, !goal, !passives stop being the only way to see state — they become a fallback for when PoE is fullscreen. The design question becomes: how much of a real companion app do we build around them?

01

Each of your four concern areas maps to a mature npm package. The only edge case is the Linux foreground-window check, and active-win handles it with an X11 / Wayland shim.

Shell choice · Electron vs Tauri vs pure CLI

Pick

Electron

Mature ecosystem, global-hotkey story is solved (uiohook-napi ships Electron prebuilds), OAuth redirect interception is one event handler. ~80MB installer cost is the only real tax.

Considered

Tauri

Smaller binaries, Rust backend. But you'd re-port every native integration (uiohook, active-win) to Rust crates with thinner docs. Not worth the rewrite for this project's scope.

Rejected

Pure CLI / headless

You lose the opportunity to surface state (items, gear, gems, goal) outside of in-game whispers — which is the single biggest UX win of the rewrite.

02

Capabilities to replicate

Pulled directly from Client.py, main.py, textUpdate.py, inputHelper.py, fileHelper.py, itemFilter.py, gggAPI.py, Options.py, ClientGui.py. Grouped by subsystem. Nothing here is invented.

A

Connection & identity

  • Connect to AP server — host, port, slot name, password; uses items_handling = 0b111 (receive all items from all sources).
  • Resolve & persist PoE character name — ctx.character_name. Set manually or via the !ap char whisper handshake (random token, user whispers char_<token>, client reads the whisperer's name from Client.txt).
  • GGG OAuth2 + PKCE — _cmd_poe_auth. Token refresh, persistent cache (currently poe_access_token.pkl).
  • Settings key — seed_name + poe-uuid + username, with "world default" fallback.
  • Version gate — warn if server's generated_version is not in BACKWARDS_COMPATIBLE_VERSIONS.
B

Filesystem integration

  • Auto-find Client.txt — find_possible_client_txt_path. Steam / standalone / registry lookup on Windows, drive scan fallback.
  • Auto-find PoE Documents folder — for filter + sound output. Overridable via /poe_documents_directory.
  • Tail Client.txt — line-by-line, dispatch to zone-change and chat-message handlers.
  • Write __ap.filter — on character validation; chain to user's base filter via Import if configured.
  • Extract bundled jingle WAVs — _ap_jingle/: Filler, Useful, Trap, ProgUseful, Progression.
  • Generate TTS WAVs — _ap_tts/, background worker, per missing location.
C

Game input (the awakened-poe-trade slice)

  • Send arbitrary text to PoE chat — via clipboard-paste-and-enter (preserving previous clipboard).
  • Global ~1s debounce — don't spam chat.
  • Chunking — async queue, split messages >500 chars, optional @<charname> prepend.
  • Foreground-window guard — only send when PoE is active; retry with backoff otherwise.
  • Global hotkeys — F12 (re-validate character / regenerate filter), F11 (dev: raise test exception).
  • Cross-platform — Windows + Linux day one; uiohook-napi handles both.
D

GGG API

  • GET /character/{name} — equipment, inventory, jewels, passives, level, class, league.
  • GET /stash/{league} and GET /stash/{league}/{tab}.
  • Rate limit — singleton request queue: 60/min + 1/sec.
  • Auto-refresh on 401 — using stored refresh token.
  • Response cache — last_response_from_api so we don't re-query on every tick.
E

Validation & Death Link

  • Zone-change trigger — parse zone-enter lines from Client.txt, re-fetch character, compare to expected state, regen filter if drift.
  • Receive Death Link — textUpdate.receive_deathlink: send /exit to kill the character.
  • Send Death Link — parse "<character> has been slain" lines for the current character.
  • DeathLink tag — managed via ctx.tags; toggleable via !deathlink whisper and /deathlink CLI.
F

Loot filter generation

  • One Show block per base item tied to remaining AP locations; skip already-checked locations.
  • Display modes (LootFilterDisplay): show_classification, hide_classification (everything styled as Progression — no spoilers), randomize_lootfilter_style (per-item random font size / text+border+bg color / minimap icon / play-effect).
  • Sound modes (LootFilterSounds): no_sound, TTS (per-item name), jingles (one WAV per classification: Filler / Useful / Trap / ProgUseful / Progression).
  • Chained base filter — user's own .filter referenced via Import, so their personal filter still applies below AP's blocks.
G

Options (slot / per-world, from Options.py)

  • Goal — complete_the_campaign / complete_act_<N> / kauri_fortress_act_6 / defeat_bosses.
  • Boss rush — number_of_bosses, bosses_available (valid keys cover Guardian / Pinnacle / Uber tiers incl. Shaper, Elder, Sirus, Maven, Exarch, Eater, Cortex variants).
  • Character — starting_character, ascendancies_available_per_class, allow_unlock_of_other_characters, usable_starting_gear, gucci_hobo_mode.
  • Item pool — progressive gear on/off, gear-upgrade tier gating, flasks, links (support-gem slots), passive skill points, leveling-up as locations, skill gems, support gems. Most have a <X>_per_act minimum.
  • Client options — loot_filter_sounds, loot_filter_display, enable_tts, tts_speed.
  • Presets — Quick Sync, Guardian / Pinnacle / Uber Boss Rush, Existing-Character variants, "Impossible".
H

Per-client settings (persisted)

  • tts_speed, tts_enabled
  • loot_filter_sounds, loot_filter_display
  • client_txt — path override
  • last_char — remembered character name
  • base_item_filter — user filter to chain
  • poe_doc_path — Documents folder override
  • whisper_updates — per-item whisper on/off
  • already_received_items — to suppress re-spam on reconnect

04

How this shapes the design

The biggest design leverage comes from recognizing that many of those !gear-style chat commands exist because there was no alternative. A real companion app means we can promote them from "all I can do" to "still there for when PoE is fullscreen."

01Chat commands become a fallback, not the primary surface

The UI renders the same !gear, !gems, !goal, !passives data — richer, sorted, searchable. The chat commands stay because fullscreen PoE + Alt+Tab is a real cost mid-run. But the source of truth for "where am I?" moves into the app.

02Setup goes from scavenger hunt to first-run wizard

Currently: user has to know /set_client_text_path, /poe_documents_directory, /poe_auth, /start in some order. Rewrite: wizard walks paths (auto-detected with manual override) → OAuth → test send → connect → character handshake. No prior slash-command knowledge needed.

03Filter config stops being three magic numbers

Today: loot_filter_sounds 0|1|2 + loot_filter_display 0|1|2. Rewrite: visual selector with live preview of a sample item block (with the actual font size / border / beam the player would see), plus one-click jingle preview.

04State becomes observable

Connection status, current zone, character level, Client.txt tail, AP hint feed, item-received log, boss checklist — all visible at a glance. The Python client logs these to one scrollback; we split them into dedicated panes.

05Tray presence for long sessions

Runs the background services even when the window is closed. Tray menu exposes: connection status LED, stop/start, open window, re-validate character (F12 equivalent). Matches how awakened-poe-trade behaves.

06Deathlink and whispers get a real home

Right now toggles live behind !deathlink / !whisper updates. In the app: a Settings pane with toggles + recent Death Link activity (who sent the last kill, when, from what slot).

Branching decision — blocks mocks

How ambient should the app be during a run?

This is the one question that meaningfully forks the design, and I want your call before I draw anything. Three honest directions:

  • Tray-firstSmall tray icon, HUD-style window you open on demand. Most of the time the app is invisible; chat commands still do the heavy lifting in-game. least disruption
  • Dashboard-firstAlways-open companion window on a second monitor (or PoE in windowed mode). Gear / gems / goal / feed always visible. Chat commands become rare. biggest UX upgrade
  • OverlayTransparent click-through overlay on top of PoE. Shows just the essentials (received items, goal progress, hotkey hints). Hardest to build; loudest visually. risky

My recommendation: build the Dashboard-first variant, and make "minimize to tray" polished so single-monitor users get the tray-first experience for free. Skip the overlay for v1 — it competes with awakened-poe-trade's overlay, and the shared-clipboard input trick doesn't compose well with an input-intercepting overlay.
